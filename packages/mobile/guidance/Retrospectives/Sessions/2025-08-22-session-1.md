## Key Technical Insights

- **Cloudflare Workers Security Evolution**: bcrypt is incompatible with Workers runtime due to Node.js dependencies. Successfully migrated to @noble/hashes with scrypt algorithm, maintaining equivalent security (16384 N parameter = bcrypt 10 rounds) while achieving Workers compatibility.

- **Database Architecture Flexibility**: Implemented dual-mode database factory pattern that automatically detects environment context - uses D1 bindings in Workers and libsql client locally. This pattern eliminated deployment friction between dev/prod environments.

- **Configuration Management Complexity**: Multiple configuration layers (wrangler.toml, .env files, AuthContext.js, api.js) created cascading failure points. Missing placeholder replacement in AuthContext.js caused registration failures despite correct other configurations.

- **TOML vs JavaScript Syntax**: Wrangler TOML requires table notation `[env.production.vars]` instead of JavaScript object syntax `vars = {}`. This syntax error blocked initial deployment but was easily resolved once identified.

- **Cloudflare Custom Domains**: Workers custom domains auto-configure routes but require manual DNS record creation even when Cloudflare manages the zone. Pages custom domains require separate API calls for setup.

## Challenges Encountered

- **bcrypt Compatibility**: Worker deployment failed due to bcrypt using `__dirname` and other Node.js-specific modules. Resolved by migrating to @noble/hashes library with equivalent security parameters and zero dependencies.

- **Configuration Placeholder Management**: Three separate files contained API URL configurations - wrangler.toml, mobile app configs, and AuthContext.js. AuthContext.js placeholder wasn't updated, causing registration failures. Developed systematic approach to grep for all placeholders.

- **DNS Propagation vs Route Configuration**: Cloudflare automatically configured Worker routes but Pages custom domains required manual DNS record creation. Learned distinction between route configuration (automatic) and DNS resolution (manual).

- **Multi-layered CORS Configuration**: CORS needed updating in multiple places as deployment URLs changed. Implemented pattern to include both regex for *.pages.dev and specific deployment URLs.
