## Key Technical Insights

- **CORS Debugging Strategy**: When facing "Failed to fetch" errors, test local dev against production API to isolate whether the issue is client-side (deployment) or server-side (API)
- **Dynamic Domain Detection**: Cloudflare Pages generates new deployment URLs on each deploy - hardcoded CORS entries become stale quickly
- **Production vs Development Environment Parity**: Environment variables behave differently in Expo production builds vs development, requiring runtime detection strategies
- **Cloudflare Tail Logs**: Essential for debugging - showed OPTIONS requests reaching server but no GET requests, indicating client-side CORS failure
- **Diagnostic Component Pattern**: Adding runtime debugging components (DebugInfo) provides crucial visibility into production environments where traditional debugging isn't available

## Challenges Encountered

- **"Failed to fetch" Error Masking Root Cause**: Generic browser error provided no specific details about CORS policy violations, requiring creative diagnostic approaches
- **Deployment URL Changes**: Each Cloudflare Pages deployment gets a new subdomain, breaking previously working CORS configuration without obvious indication
- **Environment Variable Loading**: Expo production builds don't load .env.production the same way as development, requiring runtime hostname detection
- **Multiple Red Herrings**: Initial focus on URL construction, environment variables, and fetch configuration when the real issue was server-side CORS policy
