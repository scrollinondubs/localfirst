## Key Technical Insights

- **Recommendation System Architecture**: The recommendation flow involves three critical components:
  1. Matching Engine: Handles user profile parsing, business scoring, and candidate selection
  2. Database Layer: Manages user preferences, consumer profiles, and business data persistence  
  3. OpenAI Integration: Generates personalized rationale based on user interests and business offerings

- **Profile Data Hierarchy**: The system has multiple user data sources with fallback logic:
  - Primary: `interview_insights` (structured JSON data)
  - Fallback: `personal_dossier` (rich profile summary with interests)
  - The matching engine needed modification to handle both formats effectively

- **Coordinate Type Coercion**: Database coordinates are stored as strings but the matching engine expects numbers. Added `parseFloat()` conversion to handle string-to-number coercion gracefully.

- **OpenAI Prompt Engineering**: Effective rationale generation requires:
  - Rich user context (personal dossier summary + interests array)
  - Detailed business data (description + services + categories)
  - Clear examples and constraints in the prompt
  - Temperature tuning (0.7) for creative but relevant responses

## Challenges Encountered

- **Generic Rationale Generation**: Initial system used rule-based logic producing boilerplate responses like "Great match for your interests and preferences." Users needed authentic, personalized explanations to believe AI analysis was real.
  - **Solution**: Implemented OpenAI-powered rationale generation with rich user and business context

- **Coordinate Parsing Issues**: Matching engine failed when coordinates were stored as strings instead of numbers in the database.
  - **Solution**: Added type-safe coordinate conversion with detailed logging for debugging

- **User Profile Data Access**: Matching engine couldn't access `personal_dossier` field because it wasn't included in the database query.
  - **Solution**: Extended database query to include all profile fields and added fallback logic

- **Authentication Context**: Initially suspected wrong user ID was being used due to mixed test data.
  - **Resolution**: Confirmed authentication was correct; issue was in UI state management timing
