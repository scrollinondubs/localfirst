## Key Technical Insights
- **Map Region Updates**: When updating map regions programmatically, it's critical to provide complete region objects with proper `latitudeDelta`/`longitudeDelta` values rather than partial updates. Partial updates can cause the map to not reorient properly.
- **Cloudflare Deployment Propagation**: Custom domain propagation can take several minutes after Pages deployment completes. Testing on the direct Cloudflare URL (e.g., `*.pages.dev`) shows immediate changes, while custom domains may lag.
- **Expo Build Dependencies**: The monorepo structure can sometimes cause build issues where Metro bundler tries to include API packages. Building from within the specific package directory (`cd packages/mobile`) resolves dependency conflicts.
- **Text Consistency**: UI text changes require checking multiple files - completion messages, screen titles, and progress indicators can all reference the same concept in different places.

## Collaboration Highlights
- **Effective Approaches**: 
  - User testing locally before production deployment caught the geocoding issue early
  - Clear communication about deployment timing prevented unnecessary production pushes
  - Systematic search for all instances of text changes ensured consistency
- **Process Improvements**:

  - Consider adding development-time alerts for partial map region updates

## Challenges Encountered
- **Map Not Reorienting**: Initially fixed geocoding coordinates but map wasn't updating visually. Root cause was partial region updates preserving old `latitudeDelta`/`longitudeDelta` values.
- **Build Process Issues**: Expo export was trying to bundle API code with mobile app, causing build failures. Resolved by running commands from correct package directory.
- **Production Propagation**: Changes appeared to not be working in production due to DNS/CDN propagation delay, initially causing confusion.
